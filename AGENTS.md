## Learned User Preferences

- When auditing or changing data paths, cross-check table and column names against `prisma/schema.prisma` and `prisma/migrations`, not assumptions from older SQL scripts.
- For security and tenancy reviews, prefer concrete file references, severities, and apply-ready fixes over high-level advice alone.
- For hub, storefront, POS, public-store, and marketing mobile work: keep desktop layouts unchanged; use dual-layout with `lg:` breakpoints (`hidden lg:block` / `lg:hidden`) — compact app-style tile hubs on small screens for hub; marketing pages use shared tokens in **`lib/utils/marketingLayout.js`** and **`MarketingSection`** (`components/marketing/layout/MarketingSection.jsx`). Marketing trust/integration copy must align with **`lib/marketing/capabilities.js`** and **`docs/MARKETING_CAPABILITY_MAP.md`** (honest Available/Roadmap labels; no unverified stats); keep marketing hero copy global, not region-specific badges. For dense marketing card grids (homepage demo store gallery, similar sections): prefer CSS-only hover via parent `group` over Framer Motion; avoid `blur-2xl`, animating to `height: auto`, and per-card React hover state; use `motion-safe:` / `motion-reduce:`.
- For automotive public storefronts: keep **`auto-parts`** parts-finder isolated; use separate vertical keys for single-brand dealership (**`vehicle-dealership`**) vs portal marketplace (**`auto-marketplace`**); do not add forum/CMS, live COE APIs, or app-download placeholders unless explicitly requested.
- For registration and onboarding: prioritize credible day-one storefront + inventory wiring across registration, hub, and public store with strict **`business_id`** tenancy; extend rich-catalog verticals via **`REGISTRATION_RICH_CATALOG_VERTICALS`** in **`lib/onboarding/registrationRichVerticals.js`** rather than one-off **`createBusiness`** branches.
- Typography and user-facing copy: use Open Sans via **`lib/fonts.ts`** and scale tokens in **`lib/utils/typography.js`**; avoid em dashes in UI copy (**`lib/utils/copyTypography.js`**, **`CopyText`**); prefer **`font-semibold`** over **`font-black`** / **`font-extrabold`** for headings; use **`MARKETING_STAT_VALUE`** / **`tabular-nums`** for KPI and price figures (Zoho/Busy-style professionalism); bulk hub typography via **`scripts/normalize-hub-typography.mjs`**, dash cleanup via **`scripts/normalize-copy-dashes.mjs`**.

## Learned Workspace Facts

- **Next.js App Router**: files with top-level **`'use server'`** export only **async** Server Actions (move sync helpers to **`lib/utils/*`**). Root **`proxy.ts`** exports **`proxy()`** (not middleware) for security headers/metadata only—tenant/business resolution stays in Node route handlers (Prisma/pg), not in the proxy file.
- **RSC / server actions → Client Components**: Prisma **`Decimal`** values are not serializable across the boundary (e.g. **`customers.credit_limit`**, **`outstanding_balance`**, **`opening_balance`**). Run payloads through **`serializeDecimalsDeep`** from **`lib/utils/serializePrismaDecimals.js`** before returning them to the client; do not rely on **`instanceof Prisma.Decimal`** alone when duplicate **`@prisma/client`** bundles can break type checks—use the helper’s decimal-like detection.
- Tenant-style isolation in this app is `business_id` scoped to `businesses`; there is no separate `tenant_id` column family across the main schema. **Forms and DB writes**: Prisma `findUnique` on tenant tables is **not** tenant-filtered by the extension—use `findFirst` with `business_id` and/or `assertEntityBelongsToBusiness` before mutations; multi-step writes should use transactions. See **`docs/DATA_INTEGRITY_AND_FORMS.md`**.
- Prisma migrations under `prisma/migrations` are the canonical database evolution path; `scripts/migrations` and `lib/db/migrations` are historical or manual—treat them as non-authoritative unless explicitly reconciled. Legacy `invoice_payments` rows may omit columns the app expects (`payment_method`, `received_by`, earlier `business_id` / soft-delete fields). Prefer Prisma **`20260606_invoice_payments_record_payment_columns`** and **`20260607_invoice_payments_received_by`**, or the idempotent SQL under `lib/db/migrations/038_invoice_payments_record_payment_columns.sql` and `039_invoice_payments_received_by.sql`, per **`docs/DATABASE_MIGRATIONS.md`**. Prisma maps `invoice_payments.received_by` / `deleted_by` as plain strings so values stay compatible with **`user`.id** (text).
- Customer AR balance truth uses **`invoice_payments`** and **`calculate_invoice_balance()`** (`InvoicePaymentService`, `PaymentModal`, API v1 invoice payments). Legacy **`payments`** + **`payment_allocations`** (`PaymentService`, `PaymentManager`) is a separate ledger path and does not reduce invoice balance unless explicitly routed through **`invoice_payments`**. Canonical payment types: **`lib/utils/paymentTypes.js`** (`receipt`/`payment`). **Purchase PO status** canon: **`lib/constants/purchaseStatus.js`** (`draft`/`ordered`/`received`/`cancelled`; **`normalizePurchaseStatus`** maps `sent`/`pending` → `ordered`) — use in Zod schemas, API routes, and PO/GRN UI.
- Hub inventory display stock comes from **`ProductService.resolveDisplayStock()`** (sum **`product_stock_locations`**, then batch/reserved qty, then variants/headline **`products.stock`**). Canonical **writes** use **`InventoryService`** (`product_stock_locations` → **`syncProductStock`**); **`batchAllocation.js`** / **`reservationManagement.js`** are unwired — FIFO/reservations live in **`InventoryService`**. Public storefront checkout (**`POST /api/storefront/[domain]/orders`**) decrements headline/variant stock via direct SQL and **bypasses** InventoryService (locations, movements, reservations). **`upsertIntegratedProductAction`** in **`lib/actions/premium/automation/inventory_composite.js`** must run stock add/remove on the **same pg client** (`invokeAddStockInTx` / `invokeRemoveStockInTx`); nested **`addStockAction`** opens another connection and can throw **Product not found** on uncommitted inserts. Placeholder serial rows must not force batch/serial mode; explicit headline **`stock`** is reconciled after batch/serial handling so inline grid edits persist.
- **Multi-country onboarding**: use **`lib/utils/regionalHelpers.ts`** (`getRegionalStandards`, `getRegistrationCountryOptions`) and **`docs/REGIONAL_STANDARDS.md`**. Registration stores `businesses.country` as the **canonical country name**; wizard/API may send **ISO or legacy aliases**. Extend **`REGIONAL_REGISTRY`** + **`ALIAS_TO_ISO`** before adding currencies in **`lib/currency/index.ts`**. Country-aware brand catalogs, payment gateways, and tax overlays resolve via **`lib/regionalMarket/`** (`getBrandsForMarket`, `getMarketFeatures`, **`domainBrandMap.js`**); hub callers use **`getDomainKnowledgeForBusiness(category, business)`** (not bare **`getDomainKnowledge(category)`**); pass **`countryIso`** into domain knowledge and run **`bun run verify:regional-market`** after catalog edits. **Runtime pack:** **`getBusinessRegionalPack`** (**`lib/utils/businessRegionalContext.js`**) merges registration, financials, and registry defaults; **`BusinessContext`** exposes **`regionalPack`** — hub/POS/print currency and tax labels must come from **`useBusiness()`**, not hardcoded PKR; public storefront currency/locale/tax via **`StoreProviders`** and **`lib/storefront/storefrontRegional.js`**; **`getPosUiConfig`** and thermal receipts use pack **`locale`**/**`taxIdLabel`**/**`defaultTaxRate`** over domain defaults. **Registration onboarding:** unified in **`lib/onboarding/*`** (`registrationStorefrontDefaults`, `registrationRichVerticals`, `registrationCategoryPresets`) + **`lib/utils/registrationSeed.js`** / **`RegistrationSeedService`**; **`createBusiness`** applies **`getDemoStorefrontProfile`**-derived storefront defaults for all domains, per-tenant **`store_payment_settings`**, then **`StorefrontSyncService.initializeStorefront`**; **`RegistrationInventoryBootstrap`** provisions primary warehouse + **`product_stock_locations`** for multi-location verticals; run **`bun run verify:registration-flow`** after edits.
- **Verticals (62+ keys)**: canonical presets live in **`lib/domainData/*.js`** → **`lib/domainKnowledge.js`**; each row should define **`intelligence`** (seasonality, peaks, perishability, lead time, etc.) for Industry Insights, Smart Restock, and AI forecasting. **`getDomainKnowledge`** merges defaults and supplies a **`name`** from the slug when the row omits it. Run **`bun run verify:domains`** after edits; see **`docs/DOMAIN_VERTICALS.md`**. Product **`domain_data`** reads/writes should resolve keys with **`normalizeKey(field)`** from **`lib/utils/domainHelpers.ts`** — normalize legacy aliases instead of dropping fields. **`REGISTRATION_RICH_CATALOG_VERTICALS`** (`lib/onboarding/registrationRichVerticals.js`) seeds starter catalogs on signup for **`auto-parts`** and **`vehicle-dealership`** only; other verticals get category shells + demo-profile storefront defaults. **Automotive storefronts (3 isolated keys):** **`auto-parts`** (integrated parts-finder hero — part/VIN/vehicle search; **`IMMERSIVE_FINDER_HERO_TYPES`**; registration seed via **`lib/dataLab/autopartsSeedCatalog.js`** + **`lib/storefront/autoPartsArchiveMap.js`** — run **`bun run verify:autoparts-seed`**), **`vehicle-dealership`** (single-brand — **`lib/storefront/autoDealership.js`**, **`dealershipShowroomProfile.js`**, **`components/storefront/sections/dealership/*`**; re-export **`isTenvoVehiclesShowroomProfile`** from **`autoDealership.js`**; buy/sell CTAs → booking/contact with owner info), **`auto-marketplace`** (portal — **`lib/storefront/autoMarketplace.js`**, **`components/storefront/sections/marketplace/*`**, aliases `car-marketplace`/`sgcarmart`). **Pharmacy elevated store:** **`lib/storefront/pharmacyStorefront.js`**, **`components/storefront/sections/pharmacy/*`** — isolated from other templates. **Furniture elevated store:** **`lib/storefront/furnitureStorefront.js`**, **`components/storefront/sections/furniture/*`** — **`furniture`** vertical, hero **`furniture-elevated`**, demo **`demo-furniture`**; archive seed via **`lib/dataLab/furnitureDemoCatalog.js`** + **`furnitureArchiveExtract.json`** (from **`archive/furctinure1.html`** / **`furtiniture2.html`**). Vehicle listings use **`products`** + **`domain_data`** (make, model, condition, bodytype, fuel, etc.); public list filters accept **`brand`**/**`body`**/**`fuel`**/**`condition`** in **`lib/actions/storefront/products.js`**; marketplace hero/COE ticker is owner text in **`settings.storefront.marketplace`** (not a live LTA API).
- **Marketing demo stores & scheduling**: homepage gallery in **`DemoStoreGallery`** + **`ThreeDLayeredCard`** (`components/ui/3d-layered-card.jsx`); metadata/heroes from **`lib/marketing/demoStoreGalleryMeta.js`** (`getDemoStoreHeroByDomain` reused in **`CommerceAndIntelligenceSection`**). Platform sales Calendly: **`lib/marketing/salesLinks.js`** (`getSalesMeetingUrl()` / **`NEXT_PUBLIC_SALES_MEETING_URL`**). Tenant meeting URLs: **`settings.storefront.booking.meetingUrl`** via **`lib/storefront/storefrontBooking.js`** / Store Settings when **`canConfigureTenantMeetingUrl`** (plan **`appointment_booking`** + booking verticals); never use platform Calendly for tenant storefront booking.
- **Market readiness / form wiring**: Hub forms should use **`lib/utils/formErrorHandler.jsx`** with **`actionSuccess` / `actionFailure`** from server actions; launch QA and scope are summarized in **`docs/MARKET_READINESS.md`** (not a substitute for end-to-end QA). Default SaaS trial is **14 days** via **`TRIAL_CONFIG`** in **`lib/config/platform.js`** (`getTrialExpiry`, Stripe checkout trial). Admin platform feature flags use `platform_feature_flags` and `platform_feature_flag_overrides`. The Prisma `feature_flags` model (`business_id`, `feature_name`) exists but is **not** wired into hub plan guards today — **per-tenant module overrides** use `businesses.settings.packaging` via **`planHasFeatureWithPackaging`** / **`getPackagingFromSettings`** in **`lib/subscription/effectivePlanAccess.js`** (`PLAN_FEATURE_TOGGLE_KEYS` in **`lib/config/plans.js`**). Limit overrides live in **`settings.limit_overrides`** (**`lib/utils/businessLimitOverrides.js`**, **`PLAN_LIMIT_OVERRIDE_KEYS`**); platform admins edit in **`PlatformAdminPanel`**, owners read-only in Settings → Billing. - **Hub mobile (below `lg`)**: tile hubs (`InventoryMobileHub`, `DashboardMobileHub`, `StorefrontMobileHub` + **`StorefrontTabShell`**); shared **`components/mobile/MobileHubPrimitives.jsx`**, form tokens **`lib/utils/formMobileStyles.js`**, FAB offset **`lib/utils/mobileLayout.js`** (`MOBILE_BOTTOM_NAV_CLASS`); **`HubMobileBottomNav`** + **`useHubMobileNav`**; child managers use **`useStorefrontEmbedded()`** inside the shell to skip duplicate headers — see **`docs/STOREFRONT_MOBILE_NAV.md`**. Alerts are split: in-app **`notifications`** + **`NotificationBell`**/SSE, inventory **`low_stock_alerts`** + dashboard KPIs, and Resend email — not a single unified pipeline.
